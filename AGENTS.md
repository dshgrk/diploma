# Правила работы ИИ в этом проекте

Эти правила действуют для всего проекта `diploma`.

## Память проекта

- Перед началом каждого запроса ИИ должен прочитать `.ai/developer-diary.md`.
- Этот файл не является обычным дневником, логом промптов или стенограммой общения.
- Его задача - хранить важную память проекта: архитектурные решения, договоренности, ограничения, названия сущностей, расположение важных файлов, рабочие команды, принятые подходы и предпочтения пользователя.
- ИИ должен использовать эту память при принятии решений, чтобы не забывать "что, как и где делать" в проекте.

## Что нужно записывать

Записывать только информацию, которая поможет будущей работе:

- архитектурные договоренности и паттерны;
- важные решения пользователя, например "архитектурно у нас такой pipeline";
- где находятся ключевые части проекта;
- какие команды запускать для разработки, тестов, миграций и старта;
- важные ограничения, запреты и предпочтения пользователя;
- незавершенные задачи, риски и технические долги;
- результаты исследования кода, если они будут полезны дальше.

## Плагины и инструменты

- Для задач, связанных с локальным браузером, просмотром интерфейса, ручной проверкой страниц, навигацией, кликами и визуальной проверкой результата, ИИ должен сам использовать `@browser-use`, когда это уместно.
- Для задач по созданию или изменению интерфейсов, страниц и веб-приложения ИИ должен опираться на возможности `@build-web-apps`, когда это уместно.
- Если задача фронтендовая и после изменений полезно проверить реальный результат в браузере, ИИ должен сделать это сам, а не оставлять проверку только пользователю.

## Проверка своей работы

- После собственных изменений ИИ должен по возможности делать короткие практические проверки результата.
- Минимальные проверки выбирать по задаче: запуск теста, сборки, линтера, открытие страницы, проверка ключевого пользовательского сценария, быстрая визуальная сверка интерфейса.
- Если полноценную проверку выполнить нельзя, ИИ должен сделать максимально доступную локальную проверку и явно это учитывать в выводах.

## Работа с дизайном

- В задачах по UI и frontend ИИ должен брать за основу уже существующий дизайн проекта и продолжать его, а не менять визуальный язык без необходимости.
- Нужно сохранять существующие паттерны: композицию, отступы, типографику, цветовую систему, стиль компонентов и поведение интерфейса, если пользователь отдельно не просит редизайн.
- Если в проекте уже есть устоявшийся стиль, новые экраны и блоки должны выглядеть как естественное продолжение текущего продукта.

## Что не нужно записывать

- Не сохранять каждый промпт пользователя целиком.
- Не вести подробную хронологию всех действий ИИ.
- Не записывать очевидные мелкие операции, временные рассуждения и шум.
- Не записывать секреты: пароли, токены, ключи API, содержимое `.env` и приватные данные.
- Если пользователь просит не сохранять конкретную информацию, ИИ должен уважать это.

## Как обновлять память

- После заметной работы ИИ должен обновить `.ai/developer-diary.md`, но только если появилась полезная долговременная информация.
- Новые важные факты добавлять в подходящий раздел, а не обязательно как новую дневниковую запись.
- Раздел "Журнал обновлений памяти" использовать кратко: только чтобы было понятно, когда и почему менялась сама память.
- Старые записи можно аккуратно уточнять и сжимать, если это делает память полезнее.
- Писать кратко, понятно и по-русски, если пользователь не просит иначе.
