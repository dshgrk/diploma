# Опис процедури інсталяції та запуску розробленого ПЗ

## 1. Призначення програмного забезпечення

Розроблене програмне забезпечення `Aurora Atelier` є вебзастосунком для продажу авторських ювелірних виробів, перегляду каталогу, оформлення замовлень, роботи з персональним кабінетом користувача та адміністрування товарів і конструктора виробів.

## 2. Системні вимоги

Для інсталяції та запуску програмного забезпечення потрібні:

- операційна система `Windows 10/11`, `Linux` або `macOS`;
- встановлений `Docker Desktop` або сумісне середовище з підтримкою `Docker Engine`;
- встановлений `Docker Compose`;
- доступ до локального файлового сховища для збереження службових даних застосунку.

## 3. Необхідні вхідні дані та підготовка середовища

Перед запуском програмного забезпечення потрібно:

1. Отримати локальну копію репозиторію проєкту.
2. Перейти в кореневу папку проєкту.
3. Створити файл `.env` на основі шаблону `.env.example`.
4. За потреби заповнити змінні середовища, що використовуються застосунком.

Основні змінні середовища:

- `APP_URL`;
- `TRUST_PROXY`;
- `GOOGLE_CLIENT_ID`;
- `SMTP_HOST`;
- `SMTP_PORT`;
- `SMTP_SECURE`;
- `SMTP_USER`;
- `SMTP_PASS`;
- `SMTP_FROM`.

## 4. Процедура інсталяції та першого запуску

Для цього проєкту основним способом інсталяції та запуску є контейнеризований запуск через `Docker Compose`.

Після переходу до кореневої папки проєкту потрібно виконати команду:

```powershell
docker compose up --build
```

У результаті буде виконано:

- побудову Docker-образу застосунку;
- запуск сервера застосунку;
- підключення контейнера до вбудованого середовища зберігання даних `SQLite`;
- відкриття мережевого порту `3000` для локального доступу.

Після успішного запуску вебзастосунок буде доступний за адресою:

```text
http://localhost:3000
```

Сторінка входу до адміністративної частини буде доступна за адресою:

```text
http://localhost:3000/admin/login
```

## 5. Заповнення демонстраційними даними

Початкові демонстраційні дані не завантажуються автоматично при кожному запуску застосунку. За потреби їх потрібно ініціалізувати окремою командою:

```powershell
docker compose --profile tools run --rm seed
```

Ця команда виконує:

- застосування міграцій бази даних;
- заповнення бази початковими демонстраційними даними.

## 6. Повторний запуск та повне очищення даних

Для повторного запуску застосунку достатньо знову виконати команду:

```powershell
docker compose up --build
```

Якщо потрібно повністю очистити локальні дані застосунку та виконати запуск з початкового стану, слід виконати:

```powershell
docker compose down -v
docker compose up --build
```

Після цього, за необхідності, демонстраційні дані повторно завантажуються окремою командою:

```powershell
docker compose --profile tools run --rm seed
```

## 7. Завершення роботи програмного забезпечення

Для коректної зупинки контейнерів застосунку потрібно виконати команду:

```powershell
docker compose down
```

## 8. Альтернативний локальний запуск без Docker

У разі потреби програмне забезпечення може бути запущене локально без контейнерів. Для цього необхідно мати встановлений `Node.js` та залежності проєкту.

Послідовність команд:

```powershell
npm install
npm run build:client
npm run start
```

Після цього застосунок також буде доступний за адресою:

```text
http://localhost:3000
```

Однак для цього проєкту рекомендованим і основним способом експлуатації є саме запуск через `Docker Compose`.

## 9. Можливі проблеми під час запуску

Під час запуску можуть виникати такі типові проблеми:

- відсутній або некоректно заповнений файл `.env`;
- не встановлено `Docker Desktop` або не запущено службу Docker;
- порт `3000` уже зайнятий іншим локальним процесом;
- демонстраційні дані не завантажені, тому каталог або службові записи виглядають порожніми.

Для усунення таких проблем слід перевірити конфігурацію середовища, доступність Docker та, за потреби, повторно виконати команди запуску й ініціалізації даних.
