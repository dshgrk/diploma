# Маршрут защиты проекта Aurora Atelier для ХАИ

Этот документ помогает быстро подготовиться к защите диплома: что показать комиссии, в каком порядке говорить, какие страницы открыть и какие вопросы могут задать.

Официальный ориентир по регламенту ХАИ: положение об экзаменационной/аттестационной комиссии указывает, что защита квалификационной работы одного студента обычно не должна превышать 30 минут. В структуре защиты отдельно указаны доклад, демонстрация эксперимента или результата и ответы на вопросы комиссии.

Источник: https://khai.edu/polozenna-pro-stvorenna-ta-organizaciu-roboti-ekzamenacijnoi-atestacijnoi-komisii

## Главная идея защиты

Короткая формула проекта:

> Aurora Atelier - это веб-система для продажи и индивидуальной сборки украшений. Пользователь может выбрать готовое изделие или собрать украшение в конструкторе, увидеть preview, получить серверно рассчитанную цену, оформить заказ и пройти mock-оплату. Администратор управляет заказами, каталогом, ассетами и данными конструктора.

Что важно подчеркнуть:

- это не просто красивая витрина, а полный e-commerce workflow;
- цена и бизнес-валидация считаются на сервере;
- база данных лежит в Docker volume и поднимается вместе с приложением;
- проект разделён на frontend features, backend modules, routes, services и документацию;
- mock payment осознанно выбран для дипломного MVP, чтобы показать контракт оплаты без риска реальных платежей.

## Тайминг доклада на 10-12 минут

### 0:00-0:40 - тема и проблема

Сказать:

> Тема работы - разработка веб-системы для бренда авторских украшений с каталогом, конструктором, корзиной, checkout, оплатой и административной обработкой заказов. Проблема в том, что обычный каталог не закрывает индивидуальные изделия: клиенту нужно заранее видеть конфигурацию, цену и итоговый состав заказа.

### 0:40-1:30 - цель и задачи

Сказать:

> Цель - создать понятную и расширяемую систему, где клиентский сценарий и админский сценарий связаны единой базой и API. Для этого были реализованы каталог, конструктор, pricing service, корзина, checkout, mock payment, личный кабинет заказов, admin orders, admin catalog и admin constructor.

### 1:30-2:40 - архитектура

Показать:

- `docs/architecture/system-overview.md`;
- схему модулей;
- `server/routes/index.js`;
- `client/src/main.jsx`.

Сказать:

> Архитектура построена как модульный монолит. Это удобно для диплома: систему легко запустить, понять и защитить, но код уже разделён по доменам. HTTP-route файлы держат контракт, service файлы держат бизнес-логику, frontend route файлы собирают страницы, а feature-папки содержат UI и helpers.

### 2:40-3:50 - база данных и Docker

Показать:

- `docker-compose.yml`;
- `db/migrations`;
- `docs/architecture/database.md`.

Сказать:

> В актуальном runtime используется SQLite, база хранится в Docker volume. Это делает запуск воспроизводимым: приложение, миграции, seed-данные и база поднимаются без ручного переноса локального файла. Для диплома это важно, потому что комиссия может проверить проект одной командой.

### 3:50-5:20 - backend и API

Показать:

- `docs/architecture/backend-api.md`;
- `server/modules/pricing/pricing.service.js`;
- `server/modules/admin-orders/admin-orders.service.js`;
- `server/modules/constructor/constructor-json.service.js`.

Сказать:

> Сервер является источником истины для цены, статусов и проверки обязательных параметров. Фронтенд может показывать preview мгновенно, но добавление в корзину и checkout проходят через серверную валидацию.

### 5:20-7:20 - публичный сайт

Открыть в браузере:

1. `/`
2. `/catalog`
3. карточку товара из каталога
4. `/constructor?type=ring&variant=ring-duet`

Сказать:

> Пользователь видит витрину, может выбрать готовое изделие или перейти в конструктор. В конструкторе тип, вариант, материал, камни и размер связаны с preview и ценой. Preview сделан в формате 1:1, чтобы изделие было видно полностью и без обрезания.

### 7:20-8:40 - корзина, checkout и оплата

Открыть:

1. `/cart`
2. `/checkout`
3. `/payment/...` после создания заказа, если есть подготовленный заказ
4. `/orders`

Сказать:

> Корзина поддерживает готовые товары и кастомные конфигурации. Checkout фиксирует контакты, доставку, согласия с офертой и возвратом. Mock payment переводит заказ из `created_pending_payment` в `confirmed`, а история статусов сохраняет аудит.

### 8:40-10:10 - административная часть

Открыть:

1. `/admin/orders`
2. `/admin/products`
3. `/admin/constructor`

Сказать:

> Администратор видит заказы, фильтры, summary, overdue-признак и детали заказа. Отдельно есть управление каталогом и конструктором: товары, материалы, ассеты, типы украшений, варианты, слоты камней и доступные камни.

### 10:10-11:20 - тесты и качество

Показать терминал:

```powershell
npm test
npm run build:client
docker compose up --build -d app
curl.exe -s -i http://localhost:3000/api/definitely-missing
```

Сказать:

> Проверка включает unit и integration tests, сборку клиента, Docker-first запуск и smoke страниц. Отдельно проверяется, что неизвестный API route возвращает JSON 404, а публичный frontend route открывается напрямую.

### 11:20-12:00 - результат

Сказать:

> Результат работы - рабочая веб-система с клиентским и административным сценарием, модульным backend, структурированным frontend, Docker-first запуском и документацией для дальнейшего развития.

## Вопросы, которые вероятнее всего зададут

### Почему SQLite, а не PostgreSQL/MySQL?

Ответ:

> Для дипломного MVP SQLite достаточно: он прост, не требует отдельного сервера и отлично работает в Docker volume. Архитектурно доступ к базе идёт через Knex и модульные сервисы, поэтому при росте проекта можно перейти на PostgreSQL без переписывания клиентской части и бизнес-сценариев.

### Почему mock payment?

Ответ:

> Цель диплома - показать полный order workflow и контракт оплаты. Реальные платежи требуют юридической и банковской интеграции, поэтому в MVP используется mock adapter. Он сохраняет аудит в `payments`, переводит заказ по статусам и может быть заменён реальным провайдером через тот же сервисный слой.

### Где защищена цена?

Ответ:

> Цена рассчитывается на сервере в pricing service. Клиентский preview нужен для UX, но сервер проверяет обязательные параметры, активность значений и итоговую сумму перед корзиной и checkout.

### Почему это не монолитный хаос?

Ответ:

> Проект - модульный монолит. Он разворачивается как один backend-сервис, но код разделён по доменам: `auth`, `catalog`, `constructor`, `pricing`, `cart`, `checkout`, `orders`, `admin-*`. Внутри модулей routes отвечают за HTTP-контракт, services - за бизнес-логику, repositories и database helpers - за данные.

### Как работает конструктор?

Ответ:

> Конструктор получает типы украшений, варианты, слоты, камни и preview assets из API. Выбор пользователя хранится в state страницы. Preview собирается слоями: база украшения плюс выбранные камни в нужных слотах. Стоимость пересчитывается сервером.

### Почему preview 1:1?

Ответ:

> Украшение визуально воспринимается как объект, который нужно видеть целиком. Поэтому canvas preview квадратный, а изображение вписывается через contain-логику. Это предотвращает гигантское масштабирование и обрезание.

### Как админ видит просроченные заказы?

Ответ:

> `overdue` не хранится как отдельный статус. Он вычисляется, если заказ находится в `in_progress` больше 14 календарных дней. Так статусная модель остаётся простой: `created_pending_payment -> confirmed -> in_progress -> completed`.

### Как проверяется безопасность?

Ответ:

> Используются серверные сессии, `httpOnly` cookie, role guard для admin endpoint-ов, проверка владельца заказа для клиентских routes и хэширование паролей. Сырые ошибки базы наружу не раскрываются.

### Что бы вы улучшили дальше?

Ответ:

> Первые улучшения: реальный payment provider, CDN/object storage для ассетов, расширенная аналитика в админке, импорт товаров, более строгая обработка изображений и e2e-тесты критических сценариев.

## На что комиссия может обратить внимание

- Соответствие документации фактическому коду.
- Работает ли Docker-first запуск.
- Есть ли разделение routes/services/features.
- Не рассчитывается ли цена только на клиенте.
- Есть ли обработка ошибок и JSON 404 для `/api/*`.
- Есть ли тесты для pricing, checkout, auth, order statuses и admin flow.
- Понятно ли, почему некоторые интеграции mock, а не production.
- Не ломается ли frontend при прямом открытии страниц.
