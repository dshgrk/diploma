# Requirements Mapping

## Связь требований и backlog

Чтобы требования из исходного документа напрямую приводили к задачам реализации, backlog ниже использует постоянные идентификаторы:

| ID | Задача |
| --- | --- |
| `T01` | Каркас проекта, окружение, миграции, seed-данные |
| `T02` | Аутентификация клиента и администратора, сессии, роли |
| `T03` | Каталог и карточка товара |
| `T04` | Конфиг конструктора и preview-контракт |
| `T05` | Сервис расчёта цены и бизнес-валидация |
| `T06` | Корзина |
| `T07` | Checkout и mock-оплата |
| `T08` | Заказы клиента |
| `T09` | Админка заказов и смена статусов |
| `T10` | Email, i18n, логирование |
| `T11` | Smoke, integration и hardening |
| `P21` | Admin CRUD товаров, материалов и option values |
| `P22` | Загрузка и проверка изображений/ассетов |
| `P23` | Фильтры `Новые`, `Просроченные` и сводка заказов |
| `P24` | Дальнейшее расширение админки |

## Что покрывает MVP, а что нет

### Полностью покрывается в MVP

- весь клиентский сценарий от регистрации до оплаты;
- просмотр своих заказов;
- административная обработка заказов;
- админское управление товарами, материалами, option values, ассетами и конструктором;
- базовые фильтры, поиск, summary и вычисляемый `overdue` для заказов;
- status workflow;
- i18n-подготовка и email-уведомления.

### Частично покрывается

- проверка изображений по формату и структуре есть в admin-потоке, но промышленная обработка изображений, CDN и автоматическая генерация размеров оставлены на развитие;
- админка покрывает основные операционные действия, но расширенная аналитика, отчёты и массовый импорт остаются backlog.
