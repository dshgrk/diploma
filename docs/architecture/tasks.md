# Implementation Tasks

## Принципы планирования

- Задачи разбиты по потоку зависимостей: `foundation -> auth -> catalog -> constructor/pricing -> cart/checkout -> payments -> orders -> admin-orders -> hardening`.
- Каждая задача считается завершённой только после кода, миграций, seed-данных и минимального тестового покрытия.
- Phase 2 не блокирует MVP и опирается на те же сущности и API-контракты.

## MVP backlog

## `T01` Каркас проекта, env, миграции, seed-данные

Цель: подготовить основу репозитория и окружения для дальнейшей разработки.

Состав работ:

- поднять структуру backend, frontend и docs;
- настроить `.env.example` и конфиг окружений;
- подключить `Express`, клиентскую раздачу статики и базовый middleware stack;
- подготовить миграции MySQL;
- создать seed-данные для администратора, товаров, материалов, типов украшений и тестовых заказов.

Критерии готовности:

- проект запускается локально одной командой;
- миграции создают схему без ручных правок;
- seed-данные позволяют открыть каталог и войти под администратором;
- структура файлов соответствует архитектурной документации.

## `T02` Аутентификация клиента и администратора

Цель: реализовать регистрацию, вход, выход и разграничение ролей.

Состав работ:

- `POST /api/auth/register`;
- `POST /api/auth/login`;
- `POST /api/auth/logout`;
- `POST /api/admin/login`;
- middleware авторизации и role guard;
- таблицы `users`, `sessions`;
- хранение сессии в `httpOnly` cookie.

Критерии готовности:

- клиент может зарегистрироваться и войти;
- администратор входит только под ролью `admin`;
- неавторизованный пользователь не получает доступ к приватным endpoint-ам;
- попытка клиента открыть админские endpoint-ы возвращает `403`.

## `T03` Каталог и карточка товара

Цель: предоставить клиенту витрину готовых изделий.

Состав работ:

- `GET /api/catalog/products`;
- `GET /api/catalog/products/:id`;
- клиентская страница каталога;
- клиентская карточка товара;
- отображение активных изображений и цен.

Критерии готовности:

- каталог показывает только активные товары;
- карточка товара открывается по ссылке и отображает галерею;
- неактивный товар не доступен в публичной зоне;
- товар можно добавить в корзину как `ready_product`.

## `T04` Конфиг конструктора и preview-контракт

Цель: построить основу конструктора и 2D layered preview.

Состав работ:

- `GET /api/constructor/config`;
- таблицы `jewelry_types`, `design_options`, `design_option_values`, `materials`;
- клиентская страница конструктора;
- модель preview-слоёв;
- мгновенное обновление визуализации при выборе значений.

Критерии готовности:

- для выбранного типа изделия подтягиваются только его опции;
- каждое активное значение может обновить preview;
- клиент не видит неактивные типы, опции и значения;
- конфиг достаточно полный, чтобы построить валидную конфигурацию изделия.

## `T05` Pricing engine и бизнес-валидация

Цель: централизовать бизнес-правила конструктора и расчёт стоимости.

Состав работ:

- `POST /api/constructor/price`;
- проверка обязательных параметров;
- проверка фиксированного списка длины;
- проверка активности выбранных значений;
- математическое округление до `0.01`;
- проверка минимальной суммы заказа на уровне сервисов checkout.

Критерии готовности:

- сервер возвращает `missing_required` для неполной конфигурации;
- невалидные значения конструктора отклоняются;
- цена идентична между preview, корзиной и checkout;
- минимум `500 грн` проверяется сервером, а не только клиентом.

## `T06` Корзина

Цель: собрать все позиции клиента в единый промежуточный слой перед заказом.

Состав работ:

- `GET /api/cart`;
- `POST /api/cart/items`;
- `PATCH /api/cart/items/:id`;
- `DELETE /api/cart/items/:id`;
- клиентская страница корзины;
- поддержка позиций `ready_product` и `custom_design`.

Критерии готовности:

- клиент видит текущую корзину после входа;
- готовые товары можно редактировать по количеству;
- кастомные позиции добавляются только после серверной валидации;
- удаление позиции сразу отражается в сумме корзины.

## `T07` Checkout и mock-оплата

Цель: перевести корзину в оплаченный заказ.

Состав работ:

- `POST /api/checkout`;
- `POST /api/payments/mock/confirm`;
- создание `orders`, `order_items`, `payments`;
- генерация `order_number`;
- фиксация согласий по оферте и возврату;
- UI checkout и экран подтверждения оплаты.

Критерии готовности:

- checkout невозможен без двух обязательных чекбоксов;
- заказ создаётся со статусом `created_pending_payment`;
- успешная mock-оплата переводит заказ в `confirmed`;
- история статусов и аудит оплаты создаются автоматически.

## `T08` Заказы клиента

Цель: дать клиенту прозрачный доступ к своим заказам после оплаты.

Состав работ:

- `GET /api/orders/me`;
- `GET /api/orders/:id`;
- экран списка заказов;
- экран деталей заказа;
- вывод истории статусов и состава заказа.

Критерии готовности:

- клиент видит только свои заказы;
- после успешной оплаты новый заказ появляется в кабинете;
- детали заказа отображают позиции, сумму, статус и историю;
- попытка открыть чужой заказ возвращает `404` или `403` по принятому контракту.

## `T09` Админка заказов и смена статусов

Цель: закрыть сценарий обработки заказа мастером.

Состав работ:

- `GET /api/admin/orders`;
- `GET /api/admin/orders/:id`;
- `PATCH /api/admin/orders/:id/status`;
- админский список заказов;
- карточка заказа;
- валидатор допустимых переходов статусов;
- вычисление признака `overdue`.

Критерии готовности:

- администратор видит все заказы;
- в карточке заказа видны контакты клиента, позиции и история;
- статус меняется только на следующий допустимый;
- заказ нельзя перевести в `in_progress` без успешной оплаты;
- `overdue` вычисляется корректно для заказов старше 14 дней в работе.

## `T10` Email, i18n, логирование

Цель: добавить обязательные сквозные возможности из требований.

Состав работ:

- email-адаптер для уведомлений о смене статуса;
- журнал `notification_logs`;
- перевод UI-строк на `uk/en`;
- сохранение `preferred_locale`;
- централизованное логирование ошибок и ключевых событий.

Критерии готовности:

- после смены статуса создаётся запись уведомления;
- интерфейс может переключаться между `uk` и `en`;
- отказ email-адаптера не откатывает изменение статуса;
- критичные действия пишутся в логи.

## `T11` Smoke, integration, hardening

Цель: довести MVP до предсказуемого и проверяемого состояния.

Состав работ:

- unit-тесты для pricing и status machine;
- integration-тесты для auth, checkout, payment, admin status flow;
- smoke-проверки UI;
- проверка индексов и ошибок доступа;
- базовая документация запуска и тестирования.

Критерии готовности:

- покрыты сценарии `UC-1`, `UC-2`, `UC-3`, `UC-4`, `UC-6`, `UC-7`;
- ключевые тесты проходят в CI или локально без ручных шагов;
- сервис корректно обрабатывает ошибки mock-payment и email;
- время ответа основных endpoint-ов соответствует целевому ориентиру `<= 3 сек` на разумной локальной нагрузке.

## Acceptance checklist для MVP

- Клиент может зарегистрироваться, войти и выйти.
- Клиент может посмотреть каталог и карточку активного товара.
- Клиент может собрать кастомное украшение в конструкторе.
- Цена в конструкторе считается сервером и возвращается с округлением до `0.01`.
- Невалидная конфигурация не попадает в корзину.
- Клиент может оформить заказ только при сумме `>= 500 грн`.
- Оба юридических чекбокса обязательны.
- После mock-оплаты заказ получает статус `confirmed`.
- Клиент видит свой заказ в кабинете.
- Администратор видит список заказов и переводит их по статусам без пропуска этапов.
- Смена статуса пишет историю и инициирует email-событие.

## Phase 2 backlog

## `P21` CRUD товаров и материалов

Состав работ:

- формы админки для создания, редактирования и деактивации товаров;
- формы управления материалами и словарями конструктора;
- валидация мультиязычных полей.

Критерии готовности:

- администратор может без SQL добавить товар и материал;
- деактивированные сущности исчезают из публичного интерфейса;
- изменения не ломают уже созданные заказы.

## `P22` Загрузка и проверка изображений

Состав работ:

- upload endpoint;
- проверка расширений `jpg/png`;
- проверка минимального разрешения `800x800`;
- сохранение метаданных изображения.

Критерии готовности:

- некорректные изображения отклоняются до сохранения;
- валидные изображения доступны каталогу и preview;
- администратор получает понятную причину отказа.

## `P23` Фильтры `Новые` и `Просроченные`

Состав работ:

- быстрый фильтр `Новые` для статуса `confirmed`;
- быстрый фильтр `Просроченные` по вычисляемому признаку;
- улучшение списка заказов в админке.

Критерии готовности:

- новые неопрацованные заказы можно открыть в один клик;
- просроченные заказы отфильтровываются без ручного поиска;
- фильтры не меняют базовую статусную модель.

## `P24` Расширение админки

Состав работ:

- дополнительные поля и действия для расширенного управления ассортиментом;
- подготовка к аналитике и отчётам;
- подготовка к реальному payment provider.

Критерии готовности:

- админка покрывает полный operational workflow;
- новые расширения используют существующие таблицы и API без ломки MVP-контрактов.
