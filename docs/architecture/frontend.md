# Frontend Architecture

## Цели фронтенда

- обеспечить быстрый и понятный путь от выбора изделия до оплаты;
- отрисовывать каталог, карточки товара, конструктор, корзину, checkout и кабинет клиента;
- отделить публичный интерфейс от административных экранов;
- поддержать локализацию `uk/en`;
- реализовать моментальное 2D layered preview без тяжёлых клиентских зависимостей.

## Технологический профиль

- `HTML5` для страниц и шаблонов;
- `CSS` с выделением токенов темы и модульных блоков;
- `JavaScript ES modules` без фреймворка;
- `fetch` для работы с REST API;
- клиентская маршрутизация не требуется, допускается многостраничный подход с общими модулями.

## Основные клиентские состояния

| Store | Содержимое | Кто использует |
| --- | --- | --- |
| `sessionStore` | текущий пользователь, роль, статус сессии | все страницы |
| `localeStore` | активная локаль `uk/en` | все страницы |
| `catalogStore` | список товаров, фильтры, пагинация | каталог |
| `constructorStore` | тип украшения, выбранные параметры, локальная цена, preview layers | конструктор |
| `cartStore` | активная корзина и счётчик позиций | хедер, корзина, checkout |
| `ordersStore` | список заказов клиента или администратора | кабинет клиента, админка |

## 2D layered preview

### Принцип

MVP использует стек абсолютных слоёв внутри контейнера preview. Каждое выбранное значение опции либо:

- меняет изображение существующего слоя;
- включает или выключает слой;
- меняет текстовую подпись для опций типа гравировки.

### Почему не canvas

- меньше объём клиентской логики;
- проще обновлять и дебажить;
- легче связать опции с предсказуемым набором изображений;
- удобно переиспользовать изображения в карточках товара и seed-данных.

### Контракт preview-слоя

Каждое значение опции конструктора, влияющее на визуализацию, должно содержать:

- `layer_key` — логический слой, например `base`, `chain`, `stone`, `engraving`;
- `asset_url` — путь к PNG/SVG-ассету слоя;
- `z_index` — порядок наложения;
- `transform_meta` — необязательные параметры позиционирования;
- `text_placeholder` — для текстовых слоёв.

### Поведение страницы конструктора

1. При открытии страницы загружается полный конфиг конструктора.
2. Пользователь выбирает тип украшения.
3. Фронтенд отображает только релевантные опции для выбранного типа.
4. При каждом изменении:
   - обновляется локальное состояние;
   - обновляется preview;
   - запускается серверный пересчёт цены с debounce.
5. Кнопка добавления в корзину остаётся неактивной, пока сервер не подтвердит валидную конфигурацию.

## Поведение экранов

### Каталог

- показывает только активные товары;
- поддерживает карточки товара с быстрым переходом в детали;
- отображает пустое состояние и техническую ошибку отдельными блоками.

### Карточка товара

- показывает галерею, цену, описание и статус доступности;
- позволяет добавить товар в корзину как готовое изделие;
- не показывает кнопки заказа для неактивного товара.

### Корзина

- объединяет `ready_product` и `custom_design` в одном списке;
- позволяет изменить количество готовых товаров;
- для кастомного изделия даёт только удаление, но не редактирование inline;
- показывает предупреждение о минимальной сумме заказа.

### Checkout

- запрашивает имя, email, телефон, адрес или выбранный способ доставки;
- содержит два обязательных чекбокса: оферта и условия возврата;
- не активирует оплату без подтверждения обоих согласий.

### Кабинет клиента

- показывает список собственных заказов;
- выводит статус, сумму, дату создания и краткий состав заказа;
- в деталях заказа отображает историю статусов и факт оплаты.

### Админка заказов

- список заказов с базовыми фильтрами по статусу;
- карточка заказа с составом, контактами клиента, историей статусов и действиями;
- кнопка смены статуса разрешена только для следующего допустимого статуса.

## Валидация и UX-состояния

- ошибки полей показываются inline и в зоне summary;
- сетевые ошибки показываются через toast и секцию повторной попытки;
- формы не должны зависать в неопределённом состоянии;
- каждый submit имеет `loading`, `success`, `error`;
- все модальные окна обязаны иметь кнопку закрытия в правом верхнем углу;
- у каждой страницы есть явный заголовок.

## i18n-подход

- значения UI-строк хранятся в `uk.json` и `en.json`;
- DOM-элементы размечаются через `data-i18n`;
- локаль пользователя берётся из:
  1. профиля пользователя, если он авторизован;
  2. локального выбора в браузере;
  3. значения по умолчанию `uk`.
- товары, материалы и опции приходят из API сразу в нужной локали.

## Доступность и совместимость

- клавиатурная навигация для форм и модалок;
- достаточный контраст ошибок и системных сообщений;
- изображения preview и каталога снабжаются `alt`;
- интерфейс проверяется в `Chrome`, `Firefox`, `Edge`.

## Нефункциональные ориентиры фронтенда

- первый интерактивный экран не должен блокироваться тяжёлыми библиотеками;
- API-запросы конструктору должны быть ограничены debounce;
- preview-ассеты загружаются лениво по типу украшения;
- страницы должны оставаться рабочими даже при недоступности email-адаптера и mock-payment статуса подтверждения.
