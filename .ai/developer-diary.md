# Память проекта для ИИ

Этот файл - не стенограмма запросов и не подробный дневник действий. Это рабочая память проекта: важные договоренности, архитектурные решения, правила, команды и заметки, которые будущий ИИ должен учитывать перед работой.

Перед началом задачи прочитай этот файл. После работы обнови его только тогда, когда появилась информация, которая реально пригодится дальше.

## Важные договоренности

- Пользователь хочет, чтобы этот файл хранил не все промпты и действия подряд, а только важные моменты проекта.
- Если пользователь сообщает архитектурное решение, паттерн или правило работы, его нужно сохранить здесь как постоянную память.
- Пример того, что нужно сохранять: "архитектурно у нас такой pipeline", выбранный подход к модулям, структура данных, правила работы с API, договоренности по UI/UX.
- Секреты, токены, пароли, ключи API и содержимое `.env` сюда не записывать.
- Для локальной браузерной проверки интерфейса, навигации и визуального результата ИИ должен сам использовать `@browser-use`, когда это подходит.
- Для задач по интерфейсам и веб-приложению ИИ должен использовать `@build-web-apps` как основной вспомогательный инструмент, когда это подходит.
- После собственных изменений ИИ должен делать мини-проверки результата: тесты, сборку, локальную браузерную проверку или другой короткий уместный чек.
- В UI-задачах ИИ не должен самовольно менять визуальный стиль проекта: нужно брать за основу существующий дизайн и продолжать его.

## Архитектурная память

- Пока специальных архитектурных решений в память не добавлено.
- Когда пользователь описывает архитектуру, pipeline, основные слои или важные связи между модулями, добавлять их сюда.
- Локальный `rate-limit` middleware отключается при `NODE_ENV=development`; лимиты запросов остаются активными только вне dev-режима.
- Актуальная рабочая версия приложения использует Docker-first runtime, `sqlite3` как data runtime и React-клиент, собранный в `public/react-app`; `docs/README.md` помечает часть старых архитектурных заметок как исторические.
- Интеграционный набор тестов расширен файлом `tests/integration/api-flows.test.js`: он покрывает анонимную сессию, регистрацию, клиентский login/logout, доступ к корзине, constructor config, checkout-валидацию, ownership для orders и admin orders.
- Интеграционные тесты используют общий экземпляр Knex/SQLite; в `tests/integration/cart-checkout.test.js` нельзя завершать общий `db.destroy()` в `afterAll`, иначе ломаются последующие test files.
- Генератор курсового отчёта находится в `.ai/build_coursework_report_v2.py`; в актуальной версии он собирает ручной ссылочный `ЗМІСТ` через внутренние bookmarks/PAGEREF, потому что стандартное поле TOC в этом шаблоне Word срабатывает нестабильно.
- Для визуальной проверки курсового отчёта на этой машине надёжнее экспортировать DOCX в PDF через Word COM (`SaveAs` в формат PDF); встроенный `render_docx.py` не отрабатывает, если локально нет LibreOffice/soffice.
- Главная витрина React-клиента в основном живёт в `client/src/main.jsx` и `client/src/styles.css`; 13.05.2026 для неё зафиксирован premium-luxury визуальный слой с упором на editorial hero, boutique-витрину и типографику `Cormorant Garamond` + `Manrope`.
- Для wide-desktop версии home hero (около `1920px`) заголовок нужно держать консервативнее по масштабу и давать hero более широкий shell, иначе длинные украинские слайды наезжают на правую advisory-card; ключевая настройка живёт в `client/src/styles.css` через desktop media-query для `.hero-shell`, `.hero-copy-panel` и `.hero-title`.
- Для luxury-направления главной ведущий accent следует держать в тоне `champagne / gold / ivory`; `sage/green` не использовать как главный визуальный акцент hero и ключевых home-секций.
- Для общего UI-pass по сайту системный accent также держать золотым не только на home, но и в каталоге, состояниях выбора, badges, active tabs, chips и внутренних CMS/admin-экранах, чтобы палитра не расходилась между публичной и рабочей частями.
- Для full-site color pass учитывать не только React-экраны, но и серверные email-шаблоны в `server/modules/notifications/notifications.service.js`: уведомления о коде подтверждения и статусе заказа тоже должны использовать ту же luxury-палитру `ivory / champagne / gold / espresso`, без возврата к старым зелёным акцентам.

## Рабочие команды

- `npm run dev` - запуск сервера через `nodemon server/index.js`.
- `npm run dev:client` - запуск Vite-клиента с `client/vite.config.js`.
- `npm run build:client` - сборка клиентской части.
- `npm run start` - запуск сервера через `node server/index.js`.
- `npm run test` - запуск тестов через `vitest run`.
- `npm run test:watch` - запуск Vitest в watch-режиме.
- `npm run db:migrate` - применить миграции Knex.
- `npm run db:rollback` - откатить последнюю миграцию Knex.
- `npm run db:seed` - запустить seed-данные Knex.
- `npm run db:reset` - полный сброс миграций, повторное применение и seed.
