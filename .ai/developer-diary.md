# Память проекта для ИИ

Этот файл - не стенограмма запросов и не подробный дневник действий. Это рабочая память проекта: важные договоренности, архитектурные решения, правила, команды и заметки, которые будущий ИИ должен учитывать перед работой.

Перед началом задачи прочитай этот файл. После работы обнови его только тогда, когда появилась информация, которая реально пригодится дальше.

## Важные договоренности

- Пользователь хочет, чтобы этот файл хранил не все промпты и действия подряд, а только важные моменты проекта.
- Если пользователь сообщает архитектурное решение, паттерн или правило работы, его нужно сохранить здесь как постоянную память.
- Пример того, что нужно сохранять: "архитектурно у нас такой pipeline", выбранный подход к модулям, структура данных, правила работы с API, договоренности по UI/UX.
- Секреты, токены, пароли, ключи API и содержимое `.env` сюда не записывать.
- Для локальной браузерной проверки интерфейса, навигации и визуального результата ИИ должен сам использовать `@browser-use`, когда это подходит.
- Для задач по интерфейсам и веб-приложению ИИ должен использовать `@build-web-apps` как основной вспомогательный инструмент, когда это подходит.
- После собственных изменений ИИ должен делать мини-проверки результата: тесты, сборку, локальную браузерную проверку или другой короткий уместный чек.
- В UI-задачах ИИ не должен самовольно менять визуальный стиль проекта: нужно брать за основу существующий дизайн и продолжать его.

## Архитектурная память

- Пока специальных архитектурных решений в память не добавлено.
- Когда пользователь описывает архитектуру, pipeline, основные слои или важные связи между модулями, добавлять их сюда.
- Локальный `rate-limit` middleware отключается при `NODE_ENV=development`; лимиты запросов остаются активными только вне dev-режима.
- Актуальная рабочая версия приложения использует Docker-first runtime, `sqlite3` как data runtime и React-клиент, собранный в `public/react-app`; `docs/README.md` помечает часть старых архитектурных заметок как исторические.
- Интеграционный набор тестов расширен файлом `tests/integration/api-flows.test.js`: он покрывает анонимную сессию, регистрацию, клиентский login/logout, доступ к корзине, constructor config, checkout-валидацию, ownership для orders и admin orders.
- Интеграционные тесты используют общий экземпляр Knex/SQLite; в `tests/integration/cart-checkout.test.js` нельзя завершать общий `db.destroy()` в `afterAll`, иначе ломаются последующие test files.
- Генератор курсового отчёта находится в `.ai/build_coursework_report_v2.py`; в актуальной версии он собирает ручной ссылочный `ЗМІСТ` через внутренние bookmarks/PAGEREF, потому что стандартное поле TOC в этом шаблоне Word срабатывает нестабильно.
- Для визуальной проверки курсового отчёта на этой машине надёжнее экспортировать DOCX в PDF через Word COM (`SaveAs` в формат PDF); встроенный `render_docx.py` не отрабатывает, если локально нет LibreOffice/soffice.
- Главная витрина React-клиента в основном живёт в `client/src/main.jsx` и `client/src/styles.css`; 13.05.2026 для неё зафиксирован premium-luxury визуальный слой с упором на editorial hero, boutique-витрину и типографику `Cormorant Garamond` + `Manrope`.
- Каталог готовых товаров теперь централизован в `db/catalog-products.js`: из этого файла seed `db/seeds/01_demo_data.js` поднимает витрину, а `scripts/generate-product-assets.js` генерирует прозрачные SVG-изображения в `public/assets/products/`. Базовая витрина на 13.05.2026 содержит 40 товаров: по 10 колец, браслетов, серёжек и подвесок.
- В `db/catalog-products.js` цены для витрины больше не стоит задавать вручную как demo-значения: актуальная логика считает их через `computePrice(definition)` от типа изделия, металла, камня, размера вставки и visual style. После studio-tier пересчёта ориентир по витрине такой: кольца ~`9310–21580`, браслеты ~`7920–17380`, серьги ~`8690–20350`, подвески ~`7390–16450`.
- Для wide-desktop версии home hero (около `1920px`) заголовок нужно держать консервативнее по масштабу и давать hero более широкий shell, иначе длинные украинские слайды наезжают на правую advisory-card; ключевая настройка живёт в `client/src/styles.css` через desktop media-query для `.hero-shell`, `.hero-copy-panel` и `.hero-title`.
- Для luxury-направления главной ведущий accent следует держать в тоне `champagne / gold / ivory`; `sage/green` не использовать как главный визуальный акцент hero и ключевых home-секций.
- Для общего UI-pass по сайту системный accent также держать золотым не только на home, но и в каталоге, состояниях выбора, badges, active tabs, chips и внутренних CMS/admin-экранах, чтобы палитра не расходилась между публичной и рабочей частями.
- Для full-site color pass учитывать не только React-экраны, но и серверные email-шаблоны в `server/modules/notifications/notifications.service.js`: уведомления о коде подтверждения и статусе заказа тоже должны использовать ту же luxury-палитру `ivory / champagne / gold / espresso`, без возврата к старым зелёным акцентам.

## Карта проекта

- Корень проекта: `C:\Users\WinShim\Desktop\diploma`.
- Локальная копия в `E:\diploma (7)\diploma` инициализирована как Git-репозиторий; `origin` привязан к `https://github.com/dshgrk/diploma.git`, основная ветка — `main`.
- Для локального Git в этой копии включён `core.hooksPath=.githooks`; hook `.githooks/post-commit` автоматически делает `git push origin <current-branch>` после каждого локального коммита. Отключение: `npm run git:auto-push:disable`, повторное включение: `npm run git:auto-push:install`.
- Для режима полного автосинка в проекте есть watcher `scripts/git-auto-sync.js`, но предпочтительный рабочий режим — не использовать его постоянно, чтобы не пушить промежуточные правки. Базовый безопасный вариант для этого репозитория: автопуш только после осознанного локального `git commit` через `.githooks/post-commit`.
- Проект называется `artisan-jewelry-mvp`: MVP веб-системы для продажи авторских украшений с конструктором дизайна.
- Документация проекта лежит в `docs/`.
- Готовые промпты для реалистичной генерации товарных packshot-изображений под все 40 seeded изделий лежат в `docs/product-image-prompts.md`.
- Для витрины готовых товаров реальные растр-изображения нужно складывать в `public/assets/products` и называть по `slug` товара (`<slug>.png`). Серверный helper `server/utils/product-image.js` проверяет, существует ли файл по `asset_path`, и если его нет, подставляет fallback по типу украшения; это убирает битые изображения, пока каталог заполняется постепенно.
- Тот же helper `server/utils/product-image.js` теперь должен использоваться не только в каталоге и заказах, но и в `server/modules/cart/cart.service.js`: корзина должна сериализовать `thumbnail_url` для ready-product позиций, иначе frontend откатывается на старые reference/fallback картинки даже при наличии новых файлов в `public/assets/products`.
- На клиенте guest-корзина (`localStorage`) тоже может хранить устаревшие `thumbnail_url`; при построении guest cart и в `CartItemPreview` для ready-product позиций нужно приоритетно восстанавливать путь по `product_slug` (`/assets/products/<slug>.png`), а не доверять старому snapshot объекта.
- Для готовых товаров появился отдельный size-pipeline. Логика размеров живёт в `client/src/ready-product.js` и `server/utils/ready-product-size.js`: ready-product позиции теперь могут хранить `configuration_json` c `size`, корзина различает одинаковые товары с разными размерами, а выбранный размер проходит через `product -> cart -> checkout -> order/account`.
- Для визуального подтверждения добавления в корзину frontend использует два слоя feedback: глобальное событие `aurora:item-added` анимирует `bag-button` в `Header`, а карточка товара показывает richer toast c количеством и выбранным размером. Если править UX добавления дальше, опираться на этот event, а не только на счётчик в бейдже.
- `custom_design` в корзине больше не должен быть "особым случаем" без control-логики: `server/modules/cart/cart.service.js` теперь поддерживает quantity и для custom items, а `client/src/main.jsx` показывает справа тот же quantity-control, плюс subtle badge `Налаштований дизайн` в control-cluster.
- Для всех позиций корзины действует единое бизнес-ограничение `MAX_CART_ITEM_QUANTITY = 100`: ready products и custom design нельзя добавить или обновить выше `100` штук ни на клиенте, ни на сервере. Если править cart UX дальше, не возвращать дублирующие бейджи для custom items — в корзине должен оставаться один понятный маркер `Налаштований дизайн`.
- Для подвесок теперь действует сквозная логика дополнительной цепочки и у ready products, и у `custom_design`: фронтенд передаёт только `chainOption`, а backend сам нормализует и сохраняет `configuration.chain = { option, length, metal, price }`, где `metal` всегда равен металлу подвески. Цена цепочки считается только на backend через `server/utils/pendant-chain.js` и уже в таком виде проходит через `cart -> checkout -> order -> account/admin`.
- У ready products подвески больше не используют старую псевдо-size логику для длины цепочки: `server/utils/ready-product-size.js` и `client/src/ready-product.js` сравнивают ready-конфигурации по `size` и `chain.option`, чтобы одна и та же подвеска с разной комплектацией не склеивалась в одну позицию корзины.
- Клиентская часть лежит в `client/`.
- Серверная часть лежит в `server/`.
- База данных и связанные материалы лежат в `db/`.
- Скрипты проекта лежат в `scripts/`.
- Тесты лежат в `tests/`.
- Правила для ИИ лежат в `AGENTS.md`.
- Эта память лежит в `.ai/developer-diary.md`.

## Рабочие команды

- `npm run dev` - запуск сервера через `nodemon server/index.js`.
- `npm run dev:client` - запуск Vite-клиента с `client/vite.config.js`.
- `npm run build:client` - сборка клиентской части.
- `npm run start` - запуск сервера через `node server/index.js`.
- `npm run test` - запуск тестов через `vitest run`.
- `npm run test:watch` - запуск Vitest в watch-режиме.
- `npm run db:migrate` - применить миграции Knex.
- `npm run db:rollback` - откатить последнюю миграцию Knex.
- `npm run db:seed` - запустить seed-данные Knex.
- `npm run db:reset` - полный сброс миграций, повторное применение и seed.
